# Company Naming System — Complete Methodology v3

## Overview

A naming system built from Lexicon Branding methodology (the firm behind Vercel, BlackBerry, Azure, Sonos, Windsurf, Impossible Burger), startup pattern analysis, phonetic heuristics, trademark logic, and domain strategy.

The goal is to produce names that are:
- distinctive
- memorable
- scalable
- pronounceable
- searchable
- brandable over a 10-year horizon

This methodology is designed for company names first, but it can be adapted for product names, feature names, agent names, API names, and internal codenames.

**Core sequence: Identify → Invent → Implement**

## The Three-Phase Framework

### Phase 1: Identify
Define the brief completely before generating anything. Use the Diamond Framework to reframe naming from "find a word" to "define the desired customer experience."

**Diamond Framework (four vertices):**
- **Win (top):** What defines victory for this brand?
- **Have (right):** Current advantages to build from
- **Need (bottom):** What's still required to win
- **Say (left):** The one message the name must carry

Without a sharp brief, high-volume generation produces high-volume mediocrity.

### Phase 2: Invent
Generate 60–100+ candidates across three cognitive perspectives before any evaluation begins.

**The Three Cognitive Perspectives:**

**Perspective A — The Insider**
Knows the product, the audience, the strategy. Generates from inside the brief.

**Perspective B — The Competitor**
Ignores the brief. Asks: what name would make this company's competitors most envious? Cognitive distance from the product produces names that feel category-owning rather than category-descriptive.

**Perspective C — The Adjacent Category**
Pretends to name something in a completely different industry (luxury goods, aerospace, film, fashion). Brings only the emotional register back to the product. This is how "Windsurf" and "BlackBerry" emerged — from teams working on unrelated concept clusters.

**Critical rule: Separate generation from evaluation.** Judgment during generation produces smaller, safer, more forgettable sets. Generate all candidates first. Evaluate after.

### Phase 3: Implement
Screen → Score → Conflict check → Domain strategy → Final recommendation

Domain is handled here — as a logistics problem. A great name with no .com has several paths forward: alternate TLD, variant domain (`get[name].com`), or domain purchase ($5K–$30K is rational for a name that compounds in value for decades). Never sacrifice a great name for a mediocre one because of domain anxiety.

## Naming Surfaces

Before generating names, define **what layer** is being named. Different surfaces need different rules.

| Surface | Goal | Naming Bias |
|---------|------|-------------|
| **Company name** | Long-term brand, investor-facing, category-scalable | Abstract, invented, compound |
| **Product name** | More specific, easier to understand quickly | Compound, evocative, semi-descriptive |
| **Feature name** | Clear and useful inside product UI | Descriptive, short, low ambiguity |
| **API / SDK name** | Developer clarity and ecosystem fit | Technical, literal, stable |
| **Agent / assistant name** | Personality, recall, friendliness | Warm, human, metaphorical |
| **Internal codename** | Fast coordination, low polish needed | Mythic, playful, shorthand |

### Rule
- Never use the same naming standard for every layer.
- A great company name can be too abstract for a feature.
- A great feature name can be too literal for a company.

## The 8 Name Types

| # | Type | Description | Examples | Trademark Strength |
|---|------|-------------|----------|-------------------|
| 1 | **Invented** | Completely new word, no prior meaning | Kodak, Xerox, Google, Figma, Spotify | Strongest (Fanciful) |
| 2 | **Abstract / Arbitrary** | Real word used in an unrelated context | Stripe, Apple, Shell, Target, Linear | Strong |
| 3 | **Metaphoric / Evocative** | Suggests qualities through image, symbol, or mythology | Amazon, Nike, Palantir, Atlas, Oracle | Strong |
| 4 | **Compound / Blended** | Two words merged or blended | Dropbox, Airbnb, Coinbase, Snapchat, BlackBerry | Moderate-Strong |
| 5 | **Descriptive** | Directly describes the product or function | PayPal, OpenAI, Salesforce, Cloudflare | Weak |
| 6 | **Acronym / Initialism** | Letters from a phrase | IBM, BMW, SAP, KFC | Moderate |
| 7 | **Founder / Eponym** | Based on a founder or figure | Tesla, Disney, Ford, Heroku | Moderate |
| 8 | **Foreign Word** | Borrowed or adapted from another language | Samsung, Lego, Asana | Moderate-Strong |

### Notes
- **Invented** names maximize ownability but take more work to socialize.
- **Abstract** names are common among great tech companies because they scale well.
- **Compound** names create a "1+1=3" effect — richer than either root alone, and often easier to domain.
- **Descriptive** names are risky — they reduce future flexibility and weaken trademark position.

## Semantic Generation Lenses

The taxonomy above defines **form**. The next layer is **meaning source**.

| Lens | What It Names | Example Directions |
|------|---------------|-------------------|
| **Problem** | Pain, friction, chaos, waste | Noise, drift, lag, clutter |
| **Outcome** | Speed, control, clarity, trust, growth | Signal, momentum, lift, focus |
| **Identity** | Who the user becomes | Builder, operator, maker, analyst |
| **Mechanism** | How the product works | Sync, relay, graph, stream, forge |
| **Metaphor** | Symbolic story or object | Atlas, loom, beacon, prism, forge |
| **Worldview** | Belief system or feeling | Modern, calm, exact, sovereign |
| **Category adjacency** | Nearby concepts without being literal | Stripe for payments, Plaid for fintech |

### Rule
- Names should be **meaning-adjacent**, not fully literal.
- If the product solves a painful problem, the best names speak to **outcome** or **identity**, not the pain itself.

## Letter Psychology and Sound Design

Names are not neutral. Sound shapes perception — and these patterns hold across cultures and languages.

| Letter / Sound | Psychological Effect | Example Names |
|----------------|---------------------|---------------|
| **V** | Most vibrant and alive — chosen deliberately for Vercel | Vercel, Vimeo, Vivid |
| **B** | Stable, trustworthy, grounded | Basecamp, Bench, BlackBerry |
| **Z** | Distinctive, modern, energetic | Azure, Zoom, Zappos |
| **X** | Technical, futuristic, unknown | SpaceX, Xerox, Xero |
| **Hard consonants** (B, T, K, P) | Force, clarity, confidence | Stripe, Block, Snap |
| **Soft sounds** (S, L, M) | Flow, calm, friendliness | Loom, Sonos, Slack |
| **Open vowels** (O, A) | Warm, broad, human | Canva, Sonos, Lasso |
| **Short vowels** | Punchy, memorable | Stripe, Slack, Tick |
| **"-ly" suffix** | Familiar modern software tone | Grammarly, Clearly |
| **"-ify" suffix** | Action-oriented, productized | Spotify, Shopify, Amplify |
| **"-io" ending** | Technical, startup-native | Twilio, Curio |
| **"-a" ending** | Soft, approachable, abstract | Figma, Asana |

### Practical Application
- Fast + technical: hard consonants + short vowels
- Friendly + broad-market: soft consonants + open vowels
- Premium + abstract: fewer syllables, cleaner endings
- Enterprise + trustworthy: B, D, hard stops, no gimmicks

## Structural Name Creation Methods

### 1. Root Word Mutation
Start from a concept root and compress or twist it.
- `measure → measr → mezr → mezra`
- `vector → vektor → vecto → vecta`

### 2. Portmanteau / Blending
Combine two roots.
- `signal + forge → Sigforge`
- `pixel + lab → Pixlab`

### 3. Collision-Based Compounding
Join two semantically unrelated but emotionally aligned words. This produces the strongest compound names.
- `Wind + Surf` (flow concepts applied to tech)
- `Black + Berry` (keyboard keys + organic)
- `Power + Book` (device + knowledge)

### 4. Vowel Removal / Compression
Remove vowels to create a tighter mark.
- `cluster → clstr`
- `lumen → lmn`

### 5. Prefix / Suffix Matrix
Move beyond ad-hoc affixing to systematic combination. Define N prefixes and M suffixes from a semantic territory, then test all N×M combinations. This is how Lexicon generates volume without losing intentionality.
- Example prefixes: `re-`, `omni-`, `hyper-`, `meta-`, `neo-`, `ver-`, `vel-`, `ac-`, `str-`, `cel-`, `rel-`
- Example suffixes: `-ly`, `-fy`, `-io`, `-lab`, `-base`, `-grid`, `-el`, `-on`, `-ix`, `-um`, `-a`, `-en`
- **Lexicon's Vercel process:** 55 prefixes × 102 suffixes drawn from speed/reliability concepts in Greek, Latin, and Sanskrit → 60 candidates. Repeated across 3 semantic territories over 2 weeks before a winner emerged.

### 6. Phonetic Shift
Adjust spelling while preserving sound.
- `lift → lyft`
- `clear → kleer`

### 7. Classical Roots
Greek / Latin roots create technical abstraction.
- `lumen`, `nexus`, `vertex`, `aether`, `nova`, `axis`

### 8. Real-Word Theft
Use a real word from the wrong category.
- `Stripe`, `Plaid`, `Notion`, `Linear`, `Ramp`

### 9. Foreign-Language Mining
Pull from other languages and adapt.
- Useful when the sound and legal profile still work in the target market.

### 10. Pattern Templates
Use repeatable phonetic templates:
- `CVCVC` → Figma, Asana style
- `CCVCV` → Stripe-adjacent rhythm
- `CVRA` / `VRA` / `LVO` style stems for invented words

### 11. Palindromic / Symmetric Construction
Create names that read identically forwards and backwards, or whose visual/phonetic halves mirror each other. Symmetrical structures enhance both visual distinctiveness and auditory memorability.
- `Sonos` — palindrome; mirrors itself, evokes sound waves reverberating
- `Navan` — palindrome; suggests forward movement, category-flexible
- `Kayak` — palindrome; instantly memorable, travel-associated
- `Civic` — palindrome; suggests community and reciprocity

## What Makes a Name Feel Strong

Strong names usually have most of these traits:
- Under 8 letters
- 1–2 syllables, occasionally 3
- Easy to say after hearing once
- Easy to spell after hearing once
- Emotionally legible without being literal
- Not boxed into one narrow feature
- Visually clean in a logo
- Likely to pass conflict screening
- Inspires both trust and imagination simultaneously — not just one

Weak names usually fail in one of four ways:
- Too generic
- Too descriptive
- Too awkward phonetically
- Too trend-dependent

## The Full Workflow

### Step 1: Identify — Build the Brief
1. What does the company do?
2. Who is the primary user?
3. What feeling should the name evoke?
4. What should the name not feel like?
5. What constraints matter?
6. What naming surface is this?
7. Is the brand meant to feel narrow, broad, premium, playful, or technical?

Apply the Diamond Framework:
- Win: What defines victory?
- Have: Current advantages?
- Need: What's still required?
- Say: One core message?

### Step 2: Identify — Set Generation Targets
Choose:
- 2–3 name types (prioritize Invented, Abstract, Compound)
- 2–3 semantic lenses
- 1 style mode
- Naming surface

### Step 3: Invent — Generate at Volume (Three Perspectives)
Generate 60–100 candidates across:
- **Perspective A (Insider):** 25–35 names from the brief
- **Perspective B (Competitor):** 20–30 names from competitive envy
- **Perspective C (Adjacent Category):** 15–20 names from emotional register, different industry

**Enforce diversity across:**
- Types (invented vs. abstract vs. compound)
- Sounds (hard vs. soft consonants, different endings)
- Root languages
- Lengths (4–8 letters)

**Critical rule:** Separate generation from evaluation. No judgment until all 60–100 are written.

### Step 4: Implement — First-Pass Filter
Eliminate names that are:
- Unpronounceable on first attempt
- Unspeakable without explanation
- Obviously generic
- Product-boxing (too narrow)
- Severe same-language negative meaning
- Direct same-category conflict

### Step 5: Implement — Score Survivors
Score each surviving candidate on the 10-category framework (1–5, max 50).
- 40+ = Strong shortlist
- 32–39 = Review manually
- <32 = Likely too weak

### Step 6: Implement — Conflict Screening
- Exact name Google search
- Same-category startup search
- Basic trademark screen
- Social handle collision check
- Cross-language meaning check

### Step 7: Implement — Domain Strategy
Check domain situation for each finalist. Present as logistics, not a gate.

**Resolution path for every domain situation:**
- Available at standard price → register on Cloudflare
- Taken, .ai/.io available → use alternate TLD
- Taken, variant available → `get[name].com` or similar
- Taken, worth buying → $5K–$30K purchase is rational for a great name
- Taken, check expiry → some domains are unused and near expiry

### Step 8: Implement — Shortlist and Rest
- Shortlist 3–5 names with full rationale
- Present with domain path, conflict status, score
- Recommend 48-hour rest test before final decision

### Step 9: Implement — Final Selection
The winning name should:
- Still feel distinct after repeated exposure
- Still feel scalable after imagining a pivot
- Still feel defensible after domain and trademark review
- Become more believable over time, not less

## Hard Rules vs. Soft Rules

### Hard Rules
Any violation should usually eliminate the name:
1. **Pronounceable** on first attempt
2. **Spellable** after hearing once
3. **No obvious same-category conflict**
4. **No severe negative meaning** in any target market
5. **Scales beyond one narrow feature**

### Soft Rules
These lower priority but do not automatically kill a candidate:
- .com unavailable at standard pricing (this is logistics, not naming)
- Slightly longer than ideal
- More abstract than the team is used to
- Not instantly loved on first read (this is often a *good* sign)

## Scoring Framework

| Category | Question |
|----------|----------|
| **Memorability** | Will people remember it after one exposure? |
| **Pronounceability** | Can most users say it correctly? |
| **Spellability** | Can most users spell it correctly after hearing it? |
| **Distinctiveness** | Does it stand apart from same-category brands? |
| **Brand Fit** | Does it feel right for this product and audience? |
| **Scalability** | Will it still work after expansion or pivot? |
| **Searchability** | Can it rank and be found without drowning in noise? |
| **Trademark Strength** | Is it likely to be more ownable than descriptive? |
| **International Safety** | Does it travel reasonably well across markets? |
| **Phonetic Quality** | Does the sound carry the right psychological register? |

**Thresholds:**
- 40+ = strong shortlist candidate
- 32–39 = review manually
- <32 = likely too weak

## The 10 Core Tests

1. **Cocktail Party Test** — Heard once in noise, can you remember and repeat it?
2. **Phone Test** — Can it be spelled correctly without explanation?
3. **Investor Test** — Does it sound credible to serious buyers or investors?
4. **T-Shirt Test** — Would the team proudly wear it?
5. **Google Test** — Is it searchable and not buried under unrelated results?
6. **Longevity Test** — Does it survive a 10-year company arc?
7. **International Test** — Does it avoid obvious cross-language problems?
8. **Polarization Test** — Does it create energy, not indifference?
9. **Competitor Test** — Would you be upset if a competitor launched with it?
10. **Trademark Test** — Is it likely more defensible than a descriptive label?

## Expert Principles

### The "Surprisingly Familiar" Principle
Placek's core philosophy: "The brain is lazy. It likes the familiar." The best names are "surprising, but surprisingly familiar." They use familiar linguistic building blocks — real morphemes, common phonetic patterns, recognizable roots — assembled in a novel configuration. This is why Lexicon's invented names sound natural rather than random:
- **Vercel** carries echoes of "versatile" and "excel" without being either
- **Pentium** evokes "titanium" and "premium" without saying them
- **Dasani** echoes "sanity," "san" (Latin: health), and "oasis" — all through sound, not spelling

The brain processes these names faster because the components are already in its library, but the combination is new enough to be ownable and memorable. Test every candidate: does it feel *familiar* (so it can be trusted) while feeling *novel* (so it can be owned)?

### Placek's Discomfort Principle
Great names often feel slightly wrong at first because they are unfamiliar, not because they are bad. Sonos was "not entertainment enough." Azure was "dumb." BlackBerry was "crazy." All became iconic.

### The Polarization Principle
Mild disagreement is healthy. Flat indifference is a warning sign. When a team is polarized on a name, that tension is a signal of strength.

### The 1+1=3 Compound Principle
The best compounds create a richer story than either root alone. BlackBerry is more memorable than either "black" or "berry." Windsurf carries more energy than either word alone.

### The Competitor Envy Principle
Ask: "If our competitor just launched with this name, what's your reaction?" If the answer is envy, the name is strong. This is a better test than "what do you think of this name?"

### The 1,000 Candidate Rule
The first 50 names are almost always the most generic. Quality emerges after volume. Generating 1,500–3,000 candidates (as Lexicon does professionally) is not excessive — it is the process.

### The Domain is an Area Code Principle
A great name compounds in value for decades. Domain availability is a one-time logistics problem. Never sacrifice a strong name for a mediocre one because the .com was available.

## Conflict Screening Layers

### Layer 1: Exact Match Search
- Google exact name
- Same-category startup search
- Company database search

### Layer 2: Trademark Risk
- USPTO TESS basic screen
- Same-class / adjacent-class review
- Identical or confusingly similar marks

### Layer 3: Digital Handle Collision
- X / LinkedIn / GitHub / Product Hunt / app-store naming overlap

### Layer 4: SEO Collision
- Is the term too generic?
- Is the result page dominated by another brand?

### Layer 5: Linguistic Screening
- Lexicon's standard: 20+ languages via a network of 250+ linguists
- Major target-market languages (always check first)
- Extended check: Spanish, Portuguese, Arabic, Hindi, Mandarin, Japanese, Korean, Indonesian, Malay, Turkish — these cover the largest global populations where unintended meanings cause the most damage
- Slang, profanity, awkward sound-alikes
- Phonetic similarity to sensitive words (not just exact matches)

### Rule
- Early-stage screening can be lightweight
- Final selection requires real legal review

## Domain Strategy

### Domain is Logistics, Not Naming

The right name matters far more than the exact .com. Domain conflicts are resolved through:
1. Alternate TLD (.ai, .io, .co, .dev) — entirely normal for modern startups
2. Variant pattern (`get[name].com`, `use[name].com`) — legitimate launch option
3. Domain purchase ($5K–$30K) — rational for a name that compounds in value

### Priority Order
1. Exact .com at standard price
2. Exact .ai/.io/.co at standard price (if category-appropriate)
3. Variant domain (`get[name].com`) at standard price
4. Exact .com on resale market if affordable
5. Creative combination of alternate TLD + strong brand name

### Recommended TLD Logic

| TLD | Best For | Notes |
|-----|----------|-------|
| .com | Default company brand | Strongest trust signal |
| .ai | AI companies | Normal for AI-native startups |
| .io | Developer tools | Common in technical products |
| .co | Short fallback | Better than awkward .com variants in some cases |
| .dev | Developer-facing products | Strong category signal |
| .app | Consumer/mobile products | Useful but more context-bound |

### Fallback Patterns
- `get[name].com`
- `use[name].com`
- `try[name].com`
- `[name]hq.com`
- `[name]labs.com`
- `[name]app.com`

### Rules
- Avoid hyphens
- Avoid numbers unless the number is intrinsic to the brand (e.g., v0)
- Avoid fallback domains that sound temporary or cheap

## What a Great Final Name Usually Feels Like

A great final name:
- sounds slightly more interesting than the team expected
- fits the company without describing it literally
- becomes more believable after repetition, not less
- is easy to say and hard to confuse
- has a plausible domain path (not necessarily the exact .com)
- feels like it could grow into a large company

That is the target standard for this naming system.

## When to Consider Renaming

Not every company needs a new name. But some names become ceilings. These are the triggers:

| Trigger | Signal | Example |
|---------|--------|---------|
| **Category expansion** | The name describes a product you've outgrown | TripActions → Navan (travel name, but now also expense + payments) |
| **Investment readiness** | The name confuses or alienates target investors | Zeit → Vercel (German name, English-speaking investors couldn't connect) |
| **International scaling** | The name doesn't travel across languages or cultures | Any name with negative meaning in target markets |
| **Competitive repositioning** | The name sounds like a follower, not a leader | Descriptive names that blend into the category |
| **Trademark conflict** | Legal pressure from a larger brand in the same space | Forced rename — better handled proactively |

Renaming is a strategic investment, not vanity. Vercel's rebrand coincided with a $21M Series A. Navan's revenue grew from $150M to $500M post-rebrand. When a name blocks growth, changing it is often the highest-ROI brand decision available.
